# Writing 128K programs in zt

This guide covers the practical recipes for targeting the ZX Spectrum
128 from zt. For the internals — how memory banking is simulated, how
the compiler routes data into banks, what the `.sna`/`.z80` formats
look like on disk — see [`128k-architecture.md`](128k-architecture.md)
instead.

## Quick start

Build a 128K program:

```
zt build myprog.fs -o myprog.z80 --target 128k
```

That produces a 131,182-byte `.z80` v3 snapshot that loads cleanly in
Fuse, ZEsarUX, Spectaculator, and every other mainstream emulator set
to Spectrum 128. You can also emit `.sna`:

```
zt build myprog.fs -o myprog.sna --target 128k
```

which produces a 131,103-byte Pentagon 128 SNA. Fuse will load it as
Pentagon 128 (which needs Pentagon ROMs installed); most other
emulators accept it regardless of machine type.

**Prefer `.z80` unless you specifically need `.sna`.** The `.z80` v3
format includes a hardware-type byte that tells the emulator "this is
a Spectrum 128" unambiguously. `.sna` 128K has no such field and
libspectrum defaults all 128K SNAs to Pentagon 128 — annoying unless
you have Pentagon ROMs at hand.

## Memory map

| Address | Bank | When present |
|---|---|---|
| `$0000–$3FFF` | ROM | always (16 KB of Pentagon BASIC / ZX 128 ROM) |
| `$4000–$7FFF` | RAM bank 5 | always — normal screen lives here |
| `$8000–$BFFF` | RAM bank 2 | always — zt code + stacks live here |
| `$C000–$FFFF` | RAM bank N | paged in via `$7FFD` |

Writing to `$C000–$FFFF` hits whichever RAM bank is currently paged
into slot 3. Eight RAM banks total (0–7); banks 5 and 2 are also
visible at their permanent addresses.

## The four banking primitives

zt exposes four Forth words for 128K paging:

```forth
BANK! ( n -- )          \ page RAM bank (n & 7) into $C000, preserve other bits
BANK@ ( -- n )          \ read currently paged bank (0..7)
RAW-BANK! ( n -- )      \ write full byte to $7FFD (for screen-select, lock, ROM)
128K? ( -- flag )       \ runtime detection: -1 if on 128K, 0 if 48K
```

`BANK!` is the word you'll use most. It preserves the shadow-screen
bit, the ROM-select bit, and the lock bit — only the low 3 bits change.

`RAW-BANK!` is for shadow-screen flips and similar operations where
you need to write specific bits that aren't the paged bank. Examples:

```forth
$07 raw-bank!   \ bank 7 in slot 3, normal screen visible (bit 3 = 0)
$0F raw-bank!   \ bank 7 in slot 3, shadow screen visible (bit 3 = 1)
$20 raw-bank!   \ lock paging — further $7FFD writes ignored until reset
```

`128K?` does a destructive probe — it writes sentinels to `$C000` in
banks 0 and 1, then restores whatever was there. Don't call it after
placing compile-time data in banks 0 or 1.

## Two patterns for populating banks

### Pattern A: runtime seeding (`BANK!` + `C!`)

Write Forth that pages a bank in and fills it. Works for dynamic data,
detection-gated programs, and anywhere you want `128k?` available.

```forth
: paint-bank  ( bank attr -- )
    swap bank!
    $C000 768 rot fill ;

: main
    128k? 0= if 2 border begin again then  \ halt on 48K
    $47 0 paint-bank   \ bank 0 = bright white
    $4F 1 paint-bank   \ bank 1 = bright yellow
    ...
```

See `examples/bank-rotator/main.fs` for a complete demo.

### Pattern B: compile-time placement (`in-bank` CREATE)

Put data directly into a specific bank at build time. The compiler
extends regular CREATE to take a bank context:

```forth
0 in-bank
    create level-data  $01 c, $02 c, $03 c, $04 c,
end-bank

: use-it
    0 bank!          \ page bank 0 into slot 3
    level-data $C000 + c@ ;   \ read via the paged-in address
```

The `create` emits its code shim in the main bank (always mapped) but
puts the data bytes into bank 0. At runtime, pre-paging bank 0 makes
the data accessible at `$C000 + offset`.

See `examples/bank-table/main.fs` for a complete demo.

Note: **compile-time data in bank 0 or 1 is incompatible with `128k?`**
because the probe corrupts offset 0 of those banks. Put your data in
banks 3, 4, 6, or 7, or skip runtime detection.

## Shadow-screen double buffering

The 128K machine has two screen buffers:
- **Normal screen** — bank 5, bitmap at `$4000`, attrs at `$5800`
- **Shadow screen** — bank 7, bitmap at `$C000`, attrs at `$D800` (when bank 7 is paged in)

Bit 3 of `$7FFD` selects which the ULA renders. Toggling that bit is
an atomic frame-boundary flip with zero CPU cost — the classic
double-buffer pattern:

```forth
: main
    7 bank!                \ bank 7 always in slot 3 so we can address it
    begin
        \ draw next frame into whichever buffer is currently hidden
        draw-plasma
        \ flip one bit, new frame appears on screen, old one is now hidden
        shadow-visible? @ if
            $07 raw-bank!  ( normal visible )
            0 shadow-visible? !
        else
            $0F raw-bank!  ( shadow visible )
            1 shadow-visible? !
        then
    again ;
```

See `examples/plasma-128k/main.fs` for the full demo and
`examples/shadow-flip/main.fs` for a minimal test.

## Stacks in 128K mode

Under `--target 128k`:
- Data stack defaults to `$BF00` (bank 2, always mapped)
- Return stack defaults to `$BE00` (bank 2, always mapped)

**Never put stacks at `$FF00`/`$FE00` in 128K programs.** That's in
the paged slot 3. A single `BANK!` would swap out your stack with the
new bank's contents. The CLI rejects this:

```
$ zt build foo.fs -o foo.z80 --target 128k --dstack 0xFF00
error: --dstack 0xFF00 lands in the paged slot ($C000+); 128k stacks
must live in bank 2
```

## Attribute-byte cheat sheet

Spectrum attribute byte: `F B PPP III` (flash, bright, paper, ink).

**Common pitfall:** if you set attributes but leave pixels at zero,
only the *paper* colour shows (there's no ink to draw). Writing `$47`
(bright white ink on black paper) with zero pixels produces a black
screen, not white.

| Want to see | Attribute byte |
|---|---|
| Solid bright white | `$78` |
| Solid bright red | `$50` |
| Solid bright green | `$58` |
| Solid bright yellow | `$68` |
| Solid bright cyan | `$48` |
| Solid bright magenta | `$68` |
| Bright white ink on black paper (text display) | `$47` (pixels must be set) |
| Flashing black on white | `$80 + $38 = $B8` |

Rule of thumb: if you're painting block colour without drawing
individual pixels, use an attribute where `paper` is the colour you
want and `ink` is usually black.

## Which `.z80` vs `.sna` when

| Scenario | Use |
|---|---|
| Loading in Fuse set to Spectrum 128 | `.z80` |
| Loading in Fuse set to Pentagon 128 (with Pentagon ROMs) | `.sna` or `.z80` |
| Loading in ZEsarUX | either |
| Loading in Spectaculator | either |
| Sending to an emulator that only accepts SNA | `.sna` |
| Uploading to World of Spectrum | `.z80` (convention) |

The simulator we test against treats them identically. The difference
is entirely in what the host emulator does on load.

## The `--paged-bank` flag

By default, `--target 128k` sets `$7FFD = $07` at snapshot startup:
bank 7 paged into slot 3, normal screen visible, Pentagon BASIC ROM
selected. This is the configuration that loads reliably on Fuse.

Empirical finding: snapshots with `$7FFD = $00` at startup produced
black-screen loads on Fuse-as-Pentagon. We didn't find the root cause —
probably some display-wiring state that gets properly established by
Fuse's initial `$7FFD` write during snapshot parsing when the value is
non-zero, but not when it's zero. Default `--paged-bank 7` sidesteps it.

Override if you need a different initial bank:

```
zt build foo.fs -o foo.z80 --target 128k --paged-bank 3
```

Valid range: 0..7. Doesn't affect the compiler or the output data
— only the one byte at SNA offset 49181 / Z80 offset 35.

## Minimum viable 128K program

```forth
: main
    7 0 cls           \ paper 7, ink 0, clear
    128k? 0= if 2 border begin again then
    \ ... your code ...
    begin again ;
```

That's it. Build with `zt build foo.fs -o foo.z80 --target 128k` and load.

## Troubleshooting

**Black screen after load**
- Using `.sna` in Fuse on Spectrum 128 mode? → load in Pentagon mode, or switch to `.z80`
- Passed `--paged-bank 0`? → try 7 instead, or drop the flag
- Painting attrs only with zero pixels? → use `$78` not `$47` (see cheat sheet)

**Tearing in animated output**
- Are you drawing to the currently-visible buffer? → use shadow-screen pattern
- Is your `hidden-attrs` / `shadow-visible?` state in sync with the actual `$7FFD` bit 3? → trace a few `raw-bank!` writes

**Program hangs immediately on load**
- Stack in bank 2? → use `$BF00` / `$BE00` (the defaults); CLI rejects other values
- Are you calling `128k?` after placing compile-time data in banks 0 or 1? → move data to banks 3+, or skip detection

**Fuse says "SNA could not be opened"**
- You're loading `.sna` on Spectrum 128 without Pentagon ROMs. Either install Pentagon ROMs, switch Fuse to Pentagon mode, or use `.z80`.
